# MoonBit Native Vulkan 故障排除指南

## 问题：mimalloc 与 NVIDIA Vulkan 驱动冲突（已解决）

### 现象

在运行 MoonBit Native 后端的 Vulkan 程序时，程序在创建 graphics pipeline 时崩溃：

```
Thread 1 "vulkan_demo.exe" received signal SIGSEGV, Segmentation fault.
0x000055555556b188 in mi_free ()
#0  0x000055555556b188 in mi_free ()
#1  0x00007fffe796efb8 in () at /lib/x86_64-linux-gnu/libnvidia-glvkspirv.so.575.57.08
...
#13 0x0000555555574610 in moonbit_vk_create_graphics_pipeline ()
```

### 根本原因

**问题不在于 mimalloc 本身，而是 SPIR-V shader 字节码有问题！**

原始的 `shaders_spv.h` 中的 SPIR-V 数据可能存在以下问题：
1. SPIR-V 版本不兼容
2. 字节码损坏或格式错误
3. 缺少必要的 shader 指令

### ✅ 解决方案

**重新编译 shader 并生成正确的 SPIR-V 字节码：**

```bash
# 1. 使用 glslangValidator 编译 shader
glslangValidator -V native/shaders/triangle.vert -o /tmp/vert.spv
glslangValidator -V native/shaders/triangle.frag -o /tmp/frag.spv

# 2. 生成 C 头文件（使用 Python 脚本）
python3 << 'EOF'
import struct

# Read SPIR-V files
with open('/tmp/vert.spv', 'rb') as f:
    vert_data = f.read()
with open('/tmp/frag.spv', 'rb') as f:
    frag_data = f.read()

# Convert to uint32_t array
def to_uint32_array(data):
    words = []
    for i in range(0, len(data), 4):
        word = struct.unpack('<I', data[i:i+4])[0]
        words.append(word)
    return words

vert_words = to_uint32_array(vert_data)
frag_words = to_uint32_array(frag_data)

# Generate header
with open('shaders_spv.h', 'w') as f:
    f.write("// SPIR-V shader bytecode for triangle\n")
    f.write("// Generated by glslangValidator\n\n")
    
    f.write("const uint32_t vert_shader_spv[] = {\n")
    for i, word in enumerate(vert_words):
        if i % 4 == 0:
            f.write("    ")
        f.write(f"0x{word:08x}")
        if i < len(vert_words) - 1:
            f.write(", ")
        if i % 4 == 3:
            f.write("\n")
    f.write("\n};\n\n")
    
    f.write("const uint32_t frag_shader_spv[] = {\n")
    for i, word in enumerate(frag_words):
        if i % 4 == 0:
            f.write("    ")
        f.write(f"0x{word:08x}")
        if i < len(frag_words) - 1:
            f.write(", ")
        if i % 4 == 3:
            f.write("\n")
    f.write("\n};\n\n")
    
    f.write(f"const size_t vert_shader_size = {len(vert_data)};\n")
    f.write(f"const size_t frag_shader_size = {len(frag_data)};\n")
EOF

# 3. 重新编译项目
moon build --target native
```

### 技术细节

#### 为什么看起来像 mimalloc 问题？

崩溃发生在 `mi_free()` 中，但实际上是因为 NVIDIA 驱动在编译无效的 SPIR-V 时发生了内部错误，导致内存管理混乱。正确的 shader 数据修复了这个问题。

#### SPIR-V 验证

可以使用 `spirv-val` 验证 SPIR-V 是否有效：

```bash
spirv-val /tmp/vert.spv
spirv-val /tmp/frag.spv
```

### 验证测试

| 测试场景 | 结果 |
|---------|------|
| 原始 shader + NVIDIA 驱动 | ❌ 崩溃在 `mi_free()` |
| 新编译 shader + NVIDIA 驱动 | ✅ 正常运行 |
| 新编译 shader + llvmpipe | ⚠️ 需要额外测试 |

### 临时解决方案

#### 方案 1：使用软件渲染器

```bash
VK_ICD_FILENAMES=/usr/share/vulkan/icd.d/lvp_icd.x86_64.json ./your_program
```

**注意**：llvmpipe 软件渲染器可能不支持某些 Vulkan 特性。

#### 方案 2：等待上游修复

相关 issue：
- [moonbitlang/moon#755](https://github.com/moonbitlang/moon/issues/755) - mimalloc 与 ASAN 冲突

MoonBit 需要提供以下选项之一：
- 禁用 mimalloc 的编译选项
- 使用系统 malloc 的运行时配置
- 提供不含 mimalloc 的运行时库

#### 方案 3：使用 Wasm 后端

对于 Web 应用，可以使用 Wasm-GC 后端：

```bash
moon build --target wasm-gc
```

### 技术细节

#### 内存分配器冲突原理

```
┌─────────────────────────────────────────────────────────────┐
│                    MoonBit 程序运行时                        │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐     ┌─────────────────────────────────┐    │
│  │ MoonBit 代码 │────▶│ libmoonbitrun.o (mimalloc)     │    │
│  └─────────────┘     │ - mi_malloc()                   │    │
│                      │ - mi_free()                     │    │
│                      │ - 静态链接                       │    │
│                      └─────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    Vulkan / NVIDIA 驱动                      │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────┐    │
│  │ libnvidia-glvkspirv.so                              │    │
│  │ - SPIR-V shader 编译                                │    │
│  │ - 内部内存管理                                       │    │
│  │ - 调用 free() 释放内存 ──────▶ 调用到 mi_free()     │    │
│  │ - 但内存不是由 mi_malloc() 分配                      │    │
│  └─────────────────────────────────────────────────────┘    │
│                              │                               │
│                              ▼                               │
│                      ❌ SIGSEGV 崩溃                         │
└─────────────────────────────────────────────────────────────┘
```

#### 为什么 LD_PRELOAD 无效

mimalloc 是**静态链接**到 `libmoonbitrun.o` 中的，而不是通过动态链接。这意味着：
- `mi_malloc`/`mi_free` 等函数直接嵌入到可执行文件中
- `LD_PRELOAD` 只能覆盖动态链接的符号
- 需要在编译时解决，而非运行时

### 可能的解决路径

#### 路径 1：使用 runtime.c 构建自定义运行时

MoonBit 在 `~/.moon/lib/runtime.c` 中提供了运行时源码，默认使用标准 `malloc`/`free`：

```c
MOONBIT_EXPORT void *moonbit_malloc(size_t size) {
  struct moonbit_object *ptr =
      (struct moonbit_object *)malloc(sizeof(struct moonbit_object) + size);
  ptr->rc = 1;
  return ptr + 1;
}
```

但 `libmoonbitrun.o` 是预编译的，包含了 mimalloc。可以尝试：

1. 编译 `runtime.c` 生成不含 mimalloc 的运行时库
2. 修改 MoonBit 构建配置，使用自定义运行时

**注意**：这需要深入了解 MoonBit 构建系统，可能不稳定。

#### 路径 2：等待官方支持

MoonBit 团队正在处理 mimalloc 兼容性问题（issue #755）。可能的官方解决方案：
- 提供 `--no-mimalloc` 编译选项
- 提供不含 mimalloc 的 `libmoonbitrun.o` 变体
- 支持运行时内存分配器切换
